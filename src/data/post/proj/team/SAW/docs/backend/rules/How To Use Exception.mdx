# Exception은 언제 사용하는가?

## 일반적으로

Exception은 보통 예상이 되는 오류 제어에 사용하게 됩니다.

흔히 Java에서는 File을 입출력 받을때 반드시 사용해야합니다.

```java
try {
	File file = new File("/path/to/file");
  FileInputStream stream = new FileInputStream(file);
} catch (FileNotFoundException e) {
  System.out.println(e.getCause());
}
```

이렇게 오류 제어를 강제로 하는 함수도 있는가 하면은 오류 제어를 사용자의 재량에 맡기는 경우도 있죠.

흔히 String 같이 일반적으로 사용이 가능한것들이 있습니다.

```java
public static void main(String[] args) {
	String str = null;
  System.out.println(str.length());
}
```

해당 코드를 사용하면 반드시 Null Point Exetpion(NPE)  오류가 발생합니다.

이런식으로 **예상**되는 오류를 제어하기 위해서 흔히 try-catch 를 사용하게 됩니다.

그리고 API 작성시에 흔히 많은 오류제어를 하게 되겠죠.

## 반복적인 에러 반환

하지만 백엔드에서 대부분의 에러들은 공통적으로 메세지를 반환하는경우가 많습니다.

```java
public ResponseEntity<String> foo1() {
    String x = null;
    try {
        System.out.println(x.length());
        return ResponseEntity.ok("ok!");
    } catch (NullPointerException e) {
        return ResponseEntity.status(400).body("faild...");
    }
}

public ResponseEntity<String> foo2() {
    int[] arr = null;
    try {
        System.out.println(arr[0]);
        return ResponseEntity.ok("ok!");
    } catch (NullPointerException e) {
        return ResponseEntity.status(400).body("faild...");
    }
}
```

하지만 해당 방식은 계속해서 무언가를 반환하기 위해 retrun과 객체를 반복적으로 작성을 해서 명시해줘야하는 경우가 많습니다.

그렇기 때문에 에러가 발생시 바로 ResponseEntity 같은 값을 리턴 할 수 있는 방법이 있습니다.

Restful API 설계와 규모가 있는 프로젝트에 아주 좋은 방식이죠.

물론 try-catch 로 감쌀 경우에는 원하는 방식으로 제어가 가능합니다.

해당 방식은 throws  같은 경우에만 작동하죠

```java
public ResponseEntity<String> foo1() throws NullPointerException{
    String x = null;
    System.out.println(x.length());
    return ResponseEntity.ok(x);
}

@Override
public CustomUserDetails loadUserByUsername(final String username) throws UsernameNotFoundException {
    Optional<UserEntity> optionalUser = userEntityRepository.findByUserID(username);

    /* 유저를 찾지 못할시 에러 강제 발생 */
    if (optionalUser.isEmpty()) throw new UsernameNotFoundException(username);

    UserEntity userEntity = optionalUser.get();
    USER_ROLE userRole = userRoleEntityRepository.findByUserID(userEntity.getUserID()).getUserRole();

    Collection<GrantedAuthority> grantedAuthorities = List.of(new SimpleGrantedAuthority(userRole.getValue()));

    return new CustomUserDetails(userEntity, grantedAuthorities);
}
```

# Exception 글로벌 관리하기

## 메서드에서 사용하기

X라는 컨트롤 클래스가 있다고 가정합시다.
그렇다면 이 메서드에서 어떠한 Exception이 발생하면 특정 객체를 반환합니다.

그리고 해당 컨트롤 내에서만 작동하는 Exception-Retrun이죠.

@ExceptionHandler 어노테이션만 사용해서 작동합니다.

```java
@RestController
public class X{

    @GetMapping("/example")
    public String foo() {
		    /* 커스텀 예외 발생 */
        throw new CustomException("오류 발생");
    }

    @ExceptionHandler(CustomException.class)
    public ResponseEntity<String> handleCustomException(CustomException e) {
        return ResponseEntity.status(HttpStatus.BAD_REQUEST)
                             .body(e.getMessage());
    }

    public static class CustomException extends RuntimeException {
        public CustomException(String message) {
            super(message);
        }
    }
}
```

## 글로벌하게 사용하기

메서드 내에서만 사용이 가능하다면 필터처럼 프로젝트 단위 내에서 Exception이 발생하면 반환을 제어하는 어노테이션도 있습니다.

@RestControllerAdvice or @ControllerAdvice (둘 모두 사용 가능)
@ExceptionHandler으로 제어합니다.

**SpringBoot의 필터들을 모두 거친 후(중요합니다!!!)**에 에러가 발생시, 해당 객체를 반환하게 되죠.

```java
@RestControllerAdvice
public class UserExceptionHandler {
    @ExceptionHandler(UsernameNotFoundException.class)
    public ResponseEntity<String> usernameNotFound(UsernameNotFoundException e) {
        return ResponseEntity.status(HttpStatus.NOT_FOUND)
                .body(String.format("해당 유저를 찾지 못했습니다. : %s", e.getMessage()));
	  }
}
```

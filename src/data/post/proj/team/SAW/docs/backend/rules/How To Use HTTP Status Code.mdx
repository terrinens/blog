# 해당 문서의 기술되는것들

**해당 문서는 기본적인 상황에서 HTTP가 사용되는지 기술되지 않습니다.**

각 로직의 시점에서 기술됩니다.

# 토큰 시점

## 400 BAD_REQUEST

- 변조 시도가 의심될 경우
- 유효하지 않은 서명일경우

## 401 Unauthorized

- 토큰이 만료될 경우


# 서비스 시점

사실 서비스의 시점에서는 상당히 모호한 경우가 많습니다.

서비스는 많은 로직들을 처리하기 때문에 능동적으로 사용해야할 경우가 맞습니다.

하지만 그럼에도 불구하고 기본적으로 사용되는 케이스들을 기술합니다.

## 200 OK

- 광범위하게 사용됩니다.
- 조회에 성공할 경우
- 검색에 성공할 경우

## 201 Created

- 게시글 등록, 수정, 삭제, 비활성화가 성공적으로 진행됐을 경우
- 로그인에 성공하여 토큰이 새롭게 생성됐을 경우
- cause : 로그인을 시도시에 유저의 토큰 테이블이 ‘수정’이 이뤄집니다.
그러므로 수정이 이뤄지는 201 에러코드가 맞다고 생각하여 사용되었지만…
보안에 위험 할 수 있다고 생각되어 변경할까 고려중입니다.

## 400 BAD_REQUEST

- 잘못된 요청일 경우
- 이미 존재해서 사용이 불가능할 경우 (새유저 등록, etc…)
- 광범위하게 사용

## 401 Unauthorized

- 로그인된 유저가 아닐 경우에

## 403 Forbidden

- 권한이 부족할 경우에 (유저가 관리자 전용 메서드에 접근시에)

## 404 Not Found

- DB에서 유효한 값을 찾지 못할경우
- 검색 같은 API는 값을 찾지 못할 수 도 있으므로 고려 대상이 아닙니다!

## 422 Unprocessable Entity

- 컨트롤에서 요구하는 형태는 맞으나, 서비스 단계에서 처리가 불가능할 경우
- DB의 형식에 맞지 않아 처리가 불가능한 형태일 경우

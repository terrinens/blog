# CORS, OPTION 문제

# 자세한 문제사항

## 😭 무슨 문제 사항이 있었나요?

특정 URI이 아닌 랜덤한 URI이 요청을 할 경우 CORS 문제로 요청이 정상적으로 진행되지 않는 문제가 생겼습니다.

/post/x 가 진행되지 않으면 /post/y 를 진행하면 되거나 일정 시간이 지나면 반대가 가능하게 되었습니다.

## 🧐 어디가 문제였었나요?

### 오류에서 동일한 문제점

여러가지를 테스트 하던 도중에 진행되지 않는 URI에서 동일한 현상의 오류가 나고 있다는 점을 발견하게 되었습니다.

오류가 발생하는 URI의 요청은 모두 요청이전에 메소드 설정을 OPTION으로 설정하여 보낸 이후에, 설정했던 값을 한번더 보낸다는것을요.

### CORS::OPTION 문제

CORS 정책중 OPTION 메소드를 보내는 이유는 실제로 해당 URI가 존재하는지 그리고 허가를 받기 위함이였습니다.

하지만 뭔가 이상했습니다.

Spring Boot3 에서 CORS 설정을 분명히 적절히 설정했고, Method 설정 또한 모든것을 오픈해주기 위해서 “\*” 을 사용하여 모든것을 허용했거든요.

### 커스텀 필터의 문제

여러가지를 테스트 하던 도중에 무언가 이상한점을 발견했습니다.

아무리 요청을 해도 400 에러를 반환할뿐이고 더 이상 진행이 되지 않았습니다.

그래서 한번 필터부분들을 차례로 살펴보았습니다.

그리고 이상한점을 찾아냈습니다.

OPTION 메소드로 넘어온 reqeust는 헤더의 정보값들이 극히 적다는것을 발견했습니다.

그리고 한가지 사실을 찾아낼수 있었죠.

**Access-Control-Allow-Headers 헤더를** 보내어 해당 헤더가 **허용**되어있는지 확인할뿐 상세한 헤더의 내용을 보내진 않는다는것을요.

그렇기 때문에 제가 커스텀한 ContextHolderFilter, JwtRequestFilter에 사용되는 로직하고 충돌이 발생한다는것을 알 수 있었습니다.

해당 필터들은 다음과 같은 순서를 사용하고 있었습니다.

헤더 값을 요구하고 만약에 비어있을 경우에 400 에러를 반환하는 방식이였죠.

그렇기 때문에 인증이 필요한 URI에서 OPTION 메소드로 넘어올 경우에 헤더의 값이 비어있으니 무조건적으로 400 에러를 반환했던겁니다.

해당 문제점을 찾아내고 커스텀 필터들에게 OPTION 메소드의 요청이 들어올 경우 헤더 값 추출 과정을 건너뛰는 로직을 추가하여 해당 문제점을 해결 할 수 있었습니다.
